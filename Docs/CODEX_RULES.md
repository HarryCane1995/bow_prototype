# Codex Rules

Правила для будущих изменений в проекте:

- Не делать монолитный `PlayerController`.
- Не менять несвязанные системы без необходимости.
- Не ломать существующие exported-поля.
- Все новые игровые настройки выносить в Inspector через `[Export]`.
- Перед изменениями понимать текущую структуру сцены.
- После изменений писать список изменённых файлов.
- При создании новых систем создавать отдельные модули.
- Не удалять рабочие сцены и ассеты.
- Не переименовывать публичные классы/ноды без необходимости.
- Маленькие итерации предпочтительнее больших переписываний.

## Практический чеклист

Перед правками:

- Проверить relevant `.tscn` и `.cs` файлы.
- Понять, какие ноды и exported-поля уже используются в Inspector.
- Ограничить изменение задачей пользователя.

После правок:

- Перечислить изменённые файлы.
- Указать, менялись ли сцены, ассеты или только код/документация.
- Если запуск или тест невозможен, явно написать почему.

## Правило для Inspector-полей

Если Codex добавляет новое поле, видимое в Godot Inspector через `[Export]`, он обязан:

1. Добавить понятное описание сущности поля в коде.
   - Использовать XML summary-комментарий на русском языке.
   - Описание должно объяснять, за что отвечает поле.
   - Описание должно объяснять, что произойдёт при увеличении или уменьшении значения.
   - Важно: в Godot C# XML summary не заполняет Inspector tooltip автоматически. Он нужен как документация в коде, подсказка для IDE и контекст для владельца проекта/LLM.

2. По возможности добавить `PropertyHint.Range`.
   - Для скоростей, времени, урона, дистанций, чувствительности, UI-размеров и похожих числовых параметров.
   - Диапазон должен быть безопасным и удобным для ручной настройки.

3. По возможности добавить suffix.
   - `"suffix:s"` для секунд.
   - `"suffix:m/s"` для скоростей.
   - `"suffix:px"` для UI-размеров.
   - `"suffix:deg"` для углов.
   - `"suffix:m"` для расстояний.

4. По возможности группировать поля через `[ExportGroup]` / `[ExportSubgroup]`.
   - Например: "Движение", "Прыжок", "Coyote Time", "Стрельба", "Визуал лука", "Прицел".

5. Не оставлять новые exported-поля без объяснения.
   - Если поле появляется в Inspector, оно должно быть понятно владельцу проекта без чтения всей логики класса.

6. Не использовать сомнительные или несуществующие C# tooltip-атрибуты.
   - Если Godot C# не поддерживает прямой tooltip для exported-полей, не выдумывать его.
   - Использовать XML summary, ExportGroup, PropertyHint.Range и suffix.
   - Если в Inspector tooltip показывает "No description available.", это ожидаемое ограничение Godot C# exports, а не повод добавлять несуществующий атрибут.

## Ограничение Godot C# Inspector tooltip

На текущем этапе проекта не рассчитывать, что `[Export]`-поля C# смогут получить настоящий текст в стандартном tooltip Inspector без отдельного editor tooling.

Практический стандарт проекта:

- XML summary описывает поле в исходном коде и помогает IDE/LLM.
- `[ExportGroup]` / `[ExportSubgroup]` делают Inspector читаемым.
- `PropertyHint.Range` и suffix помогают безопасно настраивать числовые значения.
- Надпись "No description available." в стандартном Inspector tooltip допустима для C# exported-полей.
- Не создавать кастомный Inspector plugin только ради tooltip, пока пользователь явно не попросит editor tooling.
