# Codex Rules

Правила для будущих изменений в проекте:

- Не делать монолитный `PlayerController`.
- Не класть object-specific interaction logic в `PlayerController`: player interaction code должен находить объект и вызывать общий контракт, а объект сам владеет поведением, анимацией, звуком и состоянием.
- Не менять несвязанные системы без необходимости.
- Не ломать существующие exported-поля.
- Все новые игровые настройки выносить в Inspector через `[Export]`.
- Перед изменениями понимать текущую структуру сцены.
- После изменений писать список изменённых файлов.
- При создании новых систем создавать отдельные модули.
- Не удалять рабочие сцены и ассеты.
- Не переименовывать публичные классы/ноды без необходимости.
- Маленькие итерации предпочтительнее больших переписываний.

## Практический чеклист

Перед правками:

- Проверить relevant `.tscn` и `.cs` файлы.
- Понять, какие ноды и exported-поля уже используются в Inspector.
- Ограничить изменение задачей пользователя.

После правок:

- Перечислить изменённые файлы.
- Указать, менялись ли сцены, ассеты или только код/документация.
- Если запуск или тест невозможен, явно написать почему.

## Правило для Inspector-полей

Inspector должен оставаться читаемым в первую очередь через структуру самих export-полей:

- понятные имена публичных `[Export]`-свойств;
- `[ExportGroup]` / `[ExportSubgroup]` для крупных блоков настроек;
- `PropertyHint.Range` для числовых значений;
- suffix для секунд, метров, градусов, скоростей и UI-размеров.

XML summary больше не обязателен над каждым `[Export]`. Оставлять комментарии стоит только там, где они объясняют неочевидный контракт: владельца системы, gameplay-правило, fallback, риск сломать сцену или причину ограничения. Комментарий должен объяснять "почему", а не пересказывать имя поля.

Нельзя использовать несуществующие C# tooltip-атрибуты. В Godot C# стандартный Inspector tooltip для exported-полей может показывать "No description available."; это допустимо и не требует кастомного Inspector plugin.
