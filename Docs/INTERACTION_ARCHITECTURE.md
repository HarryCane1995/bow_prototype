# Interaction Architecture

Главный принцип взаимодействия с объектами в BowPrototype:

> Игрок выражает намерение взаимодействовать, а объект сам знает, как на это взаимодействие ответить.

`PlayerController` не должен превращаться в центр знаний обо всём мире. Игроку не нужно знать, как открыть сундук, дверь, рычаг или терминал. Игрок должен найти интерактивный объект, проверить общий контракт и вызвать общий метод взаимодействия. Конкретный объект сам хранит своё состояние, ограничения, анимации, звуки и последствия.

## Ответственность игрока

`PlayerInteractionModule` отвечает за намерение и выбор цели:

- поиск интерактивного объекта;
- raycast или area detection;
- проверку дистанции;
- обработку input, например `interact`;
- показ prompt-текста или передачу prompt в UI;
- вызов общего метода `Interact`.

Модуль игрока не должен знать, является ли цель сундуком, дверью, рычагом или терминалом.

## Ответственность объекта

Интерактивный объект отвечает за собственное поведение:

- своё состояние;
- свои ограничения;
- свои анимации;
- свои звуки;
- последствия взаимодействия;
- возможность повторного взаимодействия.

Например, сундук сам знает, открыт он или закрыт, заблокирован ли ключом, какую анимацию проиграть и нужно ли выдать loot.

## Рекомендуемый контракт

Для обычных interact-объектов использовать общий интерфейс:

```csharp
public interface IInteractable
{
    string GetInteractionPrompt();
    bool CanInteract(PlayerController player);
    void Interact(PlayerController player);
}
```

Если в проекте позже появится другой стиль интерфейсов, можно адаптировать сигнатуры, но идея должна сохраниться: игрок вызывает общий контракт, а объект владеет конкретной логикой.

## Правильный поток

`PlayerInteractionModule`:

- видит объект с `IInteractable`;
- не проверяет `Chest`, `Door`, `Lever` или `Terminal`;
- вызывает `GetInteractionPrompt`;
- вызывает `CanInteract`;
- вызывает `Interact`.

`Chest`:

- сам решает, открыть себя или отказать;
- сам проигрывает `Open` animation;
- сам включает звук;
- сам выдаёт loot или обращается к `LootTable` / `InventoryService`.

`Door`:

- сама решает, открыта ли она;
- сама проверяет `locked` / `unlocked`;
- сама запускает анимацию открытия.

`Lever`:

- сам переключает своё состояние;
- сам вызывает нужный сигнал, событие или сервис.

## Объект не обязан быть комбайном

Объект должен владеть логикой своего взаимодействия, но это не значит, что он обязан реализовывать весь мир внутри себя.

Правильно:

- `Chest` знает, что он открывается.
- `Chest` хранит своё состояние `opened` / `locked`.
- `Chest` может обратиться к `InventoryService`, `LootTable`, `AudioManager`, `SaveSystem` или `EventBus`.
- `Chest` делегирует общие системы отдельным сервисам или модулям.

Неправильно:

- `PlayerController` содержит методы `OpenChest`, `OpenDoor`, `UseLever`, `HackTerminal`.
- `PlayerInteractionModule` делает цепочку `if object is Chest`, `if object is Door`, `if object is Lever`.
- Один центральный manager знает конкретную логику каждого объекта мира.

## Правило для будущей разработки

При добавлении нового интерактивного объекта:

- не добавлять его конкретную логику в `PlayerController`;
- не добавлять специальные проверки типа объекта в `PlayerInteractionModule`;
- создавать отдельный компонент или скрипт самого объекта;
- реализовывать общий контракт взаимодействия;
- хранить анимации, звуки, состояние и настройки рядом с объектом;
- общие системы использовать через сервисы, события или отдельные модули.

Короткая формула:

> Player expresses intent; world objects own their interaction behavior.

## Исключения и границы

Не все объекты обязаны быть `IInteractable`.

Например:

- `GrappleAnchor` используется `PlayerSlingshotGrappleModule` через отдельную систему наведения, потому что это не обычное взаимодействие по кнопке `interact`;
- projectile hitboxes могут иметь свою damage/hit логику;
- pickups могут быть `AutoPickup`, если подбираются касанием.

Но даже в этих случаях принцип сохраняется: игрок или модуль должен знать минимальный контракт, а объект должен хранить свои параметры и свою ответственность.
